> 本文档记录了所有已尝试过的实验方向、结论和失败原因。
> **目的**: 防止重复踩坑，让后续开发（人或AI）快速了解"什么试过了、为什么不行"。
>
> 创建: 2026-05-12 | 基于 V16~V22 回测脚本、ML/Minimax/Choice 实验脚本整理

## 一、回测方法论演进

### 核心问题：什么是"诚实"的回测？

从 V16→V22 的最大教训是 **回测方法论的严谨性比模型复杂度更重要**。

| 方法论问题 | V19的做法 | V20/V22的修复 | 影响 |
|-----------|----------|-------------|------|
| **收益计算** | 当日收益(T+0) | 次日收益(T+1) | T+0用未来信息，虚高15+pp |
| **评分组合** | 乘法链(boost×penalty) | 加法(score+=bonus, score-=penalty) | 乘法链放大极端值，偏斜严重 |
| **静态评分** | 全设5.0 | 从评分文件读取真实值 | 全5.0抹杀了基本面区分度 |

**结论**: V19的86.2%→V20的70.5%（下降15.7pp），才是真实水平。方法论修复后，后续所有版本都应基于V20的诚实框架。

## 三、各版本详细记录

### V16 — 迭代Optuna优化 (过拟合)
- **文件**: `backtest_v16_full_optuna.py` (已删除)
- **策略**: 按Risk级别分Phase优化(R2→R3→R45)，每Phase 500 trials
- **训练期**: 单窗口 03/01~04/24 (40天)
- **结果**: 训练90%，但扩展验证只有49%
- **失败原因**: 单窗口训练 → 严重过拟合到特定市场环境
- **教训**: **永远不要用单窗口训练，至少3个窗口联合优化**

### V17 — 多窗口联合优化 ⭐
- **文件**: `backtest_v17_robust.py` → `backtest_v17_extended.py`
- **策略**: 3窗口(A/B/C)同时优化 + 鲁棒目标函数 + L2正则化
- **目标函数**: `0.5*mean + 0.3*min + 0.2*(1-variance)` — 惩罚不稳定性
- **窗口**: 2025-11-15~01-15 / 01-15~03-01 / 03-01~04-24
- **结果**: 73.5% (扩展4.5个月验证)
- **成功因素**: 多窗口 + 鲁棒目标函数有效防止过拟合
- **不足**: 乘法评分链仍有偏斜

### V18 — 资金流Proxy因子 (无效)
- **文件**: `backtest_v18_new_factors.py`
- **策略**: V17基础上加入成交量proxy作为资金流向替代
- **新因子**: vol_ratio, vol_trend, price_vol_correlation, turn_trend
- **结果**: ~68%，不如V17
- **失败原因**: 用历史成交量pattern代替实时资金流向数据，proxy质量太差
- **教训**: **实时因子(资金流向/龙虎榜)只在实盘有用，回测用成交量proxy是无效的**

### V19 — 热门板块股票池 ⭐⭐
- **文件**: `backtest_v19_sector.py`
- **策略**: 限定AI/算力/芯片/半导体/光伏/电力 + 科创板，1939只
- **窗口**: 近3个月(02-07~05-06)，3个period联合优化
- **结果**: 86.2% (但方法论有T+0泄露)
- **核心创新**: 板块股票池比全市场有效得多
- **多期一致性**: A=86.4% B=83.3% C=89.5% (方差0.0006)
- **教训**: **股票池筛选比算法调参更重要**，选对板块事半功倍

### V20 — 方法论修复基线
- **文件**: `backtest_v20_fixed.py` + `v20_patch.py`
- **修复**: T+1收益 + 加法评分 + 真实静态评分
- **结果**: 70.5% (比V19的86.2%下降15.7pp)
- **多期**: A=72.7% B=58.3% C=86.7%
- **分析**: Period B偏弱(58.3%)，sector评分std=0.26(区分度差)
- **教训**: **诚实数据虽然不好看，但比虚假的高分有价值**

### V21 — 增强特征 (未修正方法论)
- **文件**: `backtest_v21_enhanced.py`
- **新特征**: 布林%B, ATR比率, OBV趋势, 量价确认
- **其他**: 熊市模式增强, 动态top_n(R5→1-2, R4→2-3), 4窗口训练
- **结果**: 86.2% (与V19相同)
- **问题**: 新特征是在有方法论bug的框架上测试的，无法评估真实贡献
- **教训**: **先修方法论再试新特征，否则结果不可信**

### V22 — 诚实+增强特征 ⭐ 当前最佳
- **文件**: `backtest_v22_honest.py`
- **策略**: V20诚实方法论 + V21增强特征 + 4窗口验证
- **结果**: 73.8% (比V20的70.5%提升3.3pp)
- **多期**: A=64.7% B=73.7% C=77.8% D=85.7%
- **vs历史**: 超越V17(73.5%)，是诚实方法论下的最佳
- **仍存问题**: Period A偏弱(64.7%)，可能需要更多训练数据

## 四、实验性模块记录

### 4.1 ML模型实验 (ml_backtest.py / ml_features.py / ml_train.py / ml_train_v3.py)

**尝试内容**: XGBoost / LightGBM / Logistic Regression 滚动训练

**ML Backtest V1 结果**:
- 牛市: 88% beat_idx ✅
- 熊市: 45% beat_idx ❌
- 震荡: 47% beat_idx ❌
- 整体: 55.3%

**ML Backtest V2 改进**: 加入熊市防御、风险自适应、动态TOP_N
- 整体提升有限

**结论**: **ML模型在A股短线预测中过拟合严重**。特征维度高但样本少，模型学到的是噪声不是信号。传统规则系统(当前方案)更稳定。

### 4.2 MiniMax API集成 (minimax_integration.py / minimax_feature_extensions.py)

**尝试内容**: 用MiniMax大模型API生成代码和分析

**结论**: 作为代码生成工具有价值，但作为交易信号源不可靠。大模型的"市场判断"本质上是对训练数据的回溯，没有预测能力。

**状态**: 已废弃，不集成到推荐系统。

### 4.3 Choice API (东方财富Choice终端) (choice_full_diagnosis.py / collect_choice_direct.py / collect_choice_fundamentals.py / test_choice_data*.py)

**尝试内容**: 通过Choice终端API获取高质量数据

**功能测试**:
- CSD时序K线: ✅ 可用
- 扩展指标(TURN/CHGPCT/AMOUNT): ✅ 可用
- 行业分类: ✅ 可用
- 财务数据: ✅ 可用

**问题**:
- Choice终端需要付费license，不稳定
- API调用限制多
- AKShare/东方财富免费API已能满足需求

**结论**: **Choice API数据质量好但成本高、依赖重**。当前用AKShare+东方财富免费API已足够。如果未来需要更高质量数据可重新考虑。

### 4.4 GUI兼容版 (a_share_gui_compatible.py)

**尝试内容**: 带GUI界面的选股工具

**状态**: 实验性代码，21KB的巨型单文件。功能完整但未投入使用。

**结论**: CLI+bat脚本更适合自动化场景，GUI对日常使用价值不大。

### 4.5 新数据源+新因子 (new_data_sources.py / new_factors.py)

**尝试内容**: 通过AKShare接入北向资金、个股资金流向、龙虎榜、融资融券数据

**因子设计**:
- 资金流向评分: 主力净流入趋势(5/3/1日)
- 龙虎榜评分: 上榜+机构买入
- 北向资金评分: 市场整体方向
- 融资融券评分: 余额变化

**结论**: **这些因子的代码逻辑是正确的，数据获取也work**。问题在于回测时没有历史数据(龙虎榜/资金流向无法回测)。在实盘推荐中可以考虑集成，但回测框架中无法验证其效果。

**建议**: 保留 `new_factors.py` 的因子计算逻辑，在 `real_feature_calculator.py` 或 `daily_recommender.py` 中集成实时因子。

### 4.6 最终验证报告 (final_validation_report.py)

**尝试内容**: V9d增强评分系统的对比验证

**状态**: 只适用于V9时代的验证，已被V16+的Optuna框架取代。

### 4.7 V20补丁 (v20_patch.py / v20_recommender_patch.py)

**尝试内容**: 将V20的方法论修复补丁应用到推荐器

**状态**: V22已包含所有V20修复，补丁文件已无用。

## 六、关键经验总结

### ✅ 有效的方法
1. **多窗口联合优化** — 至少3个时间窗口，用鲁棒目标函数
2. **板块股票池** — 限定热门板块比全市场选股效果好很多
3. **市场状态检测** — 牛/熊/震荡用不同策略是正确的
4. **T+1收益计算** — 防止未来信息泄露
5. **加法评分** — 比乘法链更稳定、可解释
6. **L2正则化** — 防止参数极端化
7. **行业分散** — MAX_INDUSTRY=1 防止板块集中风险

### ❌ 无效的方法
1. **ML/XGBoost/LightGBM** — A股短线过拟合严重
2. **成交量proxy代替资金流向** — proxy质量太差
3. **单窗口训练** — 90%训练→49%实盘的惨痛教训
4. **乘法评分链** — 放大极端值，结果虚高
5. **T+0收益** — 用当天数据预测当天，本质是作弊
6. **PE/PB估值过滤** — 短线无效
7. **过度保守(空仓)** — 错失行情
8. **过度分散(10只)** — 反而降低beat_idx

### ⚠️ 已知限制
1. **回测期偏短** — V22只验证了3个月，需更长时间验证
2. **Period A/B偏弱** — V22在早期窗口表现不稳定
3. **sector评分区分度差** — hot_sector_score几乎无变化(std=0.26)
4. **实时因子无法回测** — 资金流向/龙虎榜只在实盘可用
5. **推荐器评分映射** — daily_recommender的评分和回测评分还不完全一致
